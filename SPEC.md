# Duncan for Claude Code — Exhaustive Source Mapping & Spec

Source: CC v2.1.81 (`/tmp/claude-source/2.1.81/src/entrypoints/cli.js`)

## 1. Session Storage Layout

### Paths
- **Config dir**: `~/.claude/` (`d1()`)
- **Projects dir**: `~/.claude/projects/` (`lx()`)
- **Project dir**: `~/.claude/projects/<hashed-cwd>/` — `IO(cwd)` uses `UM(cwd)` to hash the cwd
- **Session file**: `<project-dir>/<session-id>.jsonl` — `Gv(sessionId)`
- **Current session**: `WY()` = `join(projectDir, sessionId + ".jsonl")`
- **Subagent transcripts**: `<project-dir>/<session-id>/subagents/<subdir>/agent-<agent-id>.jsonl` — `DW(agentId)`
- **Agent metadata**: `<project-dir>/<session-id>/subagents/<subdir>/agent-<agent-id>.meta.json`

### Session ID
- Generated by `MB8()` (likely `crypto.randomUUID()`)
- Global state `T8.sessionId` tracks current session
- `T8.parentSessionId` set via `IB8({ setCurrentAsParent: true })` (fork/subagent creation)
- **Not written to session file** — only tracked in memory and telemetry

## 2. JSONL Entry Types

### Transcript Messages (pass `mi()` / `isTranscriptMessage`)
```typescript
type: "user" | "assistant" | "attachment" | "system" | "progress"
```

Each transcript message has:
```typescript
{
  uuid: string;                    // unique ID (like pi's id)
  parentUuid: string | null;       // parent in tree (like pi's parentId)
  session_id: string;              // session ID
  type: "user" | "assistant" | "system" | "progress" | "attachment";
  timestamp: string;               // ISO timestamp
  isSidechain?: boolean;           // sidechain/subagent indicator
  parent_tool_use_id?: string | null;
  gitBranch?: string;
  teamName?: string;
  cwd?: string;                    // working directory at message time
  message: {
    role: string;
    content: ContentBlock[];
    model?: string;                // model ID (assistant messages)
    usage?: Usage;                 // token usage (assistant messages)
    id?: string;                   // API response ID (assistant messages)
  };
}
```

### Metadata Entries (non-transcript, keyed by sessionId or leafUuid)
```
type: "summary"              — compaction summary, keyed by leafUuid
type: "custom-title"         — session title, keyed by sessionId
type: "tag"                  — session tag, keyed by sessionId
type: "agent-name"           — agent name, keyed by sessionId
type: "agent-color"          — agent color, keyed by sessionId
type: "agent-setting"        — agent setting, keyed by sessionId
type: "mode"                 — session mode, keyed by sessionId
type: "worktree-state"       — worktree info, keyed by sessionId
type: "pr-link"              — PR link, keyed by sessionId (prNumber, prUrl, prRepository)
type: "file-history-snapshot" — file state at a message, keyed by messageId
type: "attribution-snapshot"  — attribution data, keyed by messageId
type: "content-replacement"   — content replacements, keyed by sessionId or agentId
type: "marble-origami-commit" — context collapse commit
type: "marble-origami-snapshot" — context collapse snapshot
```

## 3. Session Loading Pipeline

### Step 1: Read and parse JSONL — `G26(path, opts?)`
**Function**: `G26` (loadTranscriptFile)
**Location**: ~line 413253

1. **Large file optimization**: If file > `Sr8` threshold:
   - `K48(path, size)` finds the compaction boundary offset
   - Reads pre-boundary section for metadata only (summaries, titles, etc.)
   - Reads post-boundary section for transcript messages
   - If no compaction and file still > threshold, `vHY()` trims further

2. **Parse JSONL**: `mu(buffer)` — splits by newlines, JSON.parse each line

3. **Separate transcript from metadata**: For each parsed entry:
   - If `mi(entry)` (isTranscriptMessage): add to messages Map keyed by `uuid`
     - Skip ephemeral progress messages (`Ns6(type)`)
     - Strip `normalizedMessages` from progress data (save memory)
     - If entry is a compact boundary (`of(entry)`), reset leaf tracking
   - If `type === "summary"`: add to summaries Map keyed by `leafUuid`
   - If `type === "custom-title"`: add to customTitles Map keyed by `sessionId`
   - If `type === "tag"`, `"agent-name"`, `"agent-color"`, `"agent-setting"`, `"mode"`, `"worktree-state"`, `"pr-link"`: add to respective Maps
   - If `type === "file-history-snapshot"`, `"attribution-snapshot"`, `"content-replacement"`: add to respective Maps
   - If `type === "marble-origami-commit"`: add to contextCollapseCommits array
   - If `type === "marble-origami-snapshot"`: set as contextCollapseSnapshot

4. **Preserved segment relinking**: `wHY(messagesMap)` — see Section 4

5. **Find leaves**: Messages whose `uuid` is not any other message's `parentUuid`

6. **Leaf selection** (with optional pruning via `tengu_pebble_leaf_prune` feature flag):
   - Walk from each leaf up via `parentUuid` to find the first `user`/`assistant` message
   - Collect all reachable leaf UUIDs

**Returns**: `{ messages: Map, summaries: Map, customTitles, tags, agentNames, agentColors, agentSettings, modes, worktreeStates, prNumbers, prUrls, prRepositories, fileHistorySnapshots, attributionSnapshots, contentReplacements, contextCollapseCommits, contextCollapseSnapshot, leafUuids }`

### Step 2: Build message chain — `Vs6(messagesMap, leafMessage)`
**Function**: `Vs6`
**Location**: ~line 412769

1. Walk `parentUuid` chain from leaf to root
2. Cycle detection: if `uuid` already visited, break
3. Collect messages in order, reverse to get root→leaf
4. **Post-process**: `OHY(messagesMap, chain, visitedSet)` — handles tool result deduplication and orphan tool results

### Step 3: Preserved segment relinking — `wHY(messagesMap)`
**Function**: `wHY`
**Location**: ~line 412706

This is the complex compaction tree surgery. When compaction preserves some messages (`messagesToKeep`), the boundary marker stores:
```typescript
compactMetadata: {
  preservedSegment: {
    headUuid: string;     // first preserved message
    tailUuid: string;     // last preserved message  
    anchorUuid: string;   // the boundary marker's uuid
  }
}
```

Algorithm:
1. Find the **last compact boundary** on the path (`of(entry)` check)
2. If it has a `preservedSegment`:
   - Walk from `tailUuid` back to `headUuid` to identify the preserved segment
   - **Relink**: Set `headUuid.parentUuid = anchorUuid` (splice preserved segment after boundary)
   - **Relink**: For all messages that had `parentUuid === anchorUuid` (except head), set `parentUuid = tailUuid` (new messages follow after preserved segment)
   - **Zero out usage**: Set `input_tokens`, `output_tokens`, `cache_*` to 0 for all assistant messages in the preserved segment (prevent double-counting)
3. **Delete pre-boundary messages**: Remove all messages before the boundary that aren't in the preserved segment

### Step 4: Post-process chain — `OHY(messagesMap, chain, visited)`
**Function**: `OHY`
**Location**: ~line 412783

Handles edge cases with tool results and assistant messages:
1. For each assistant message with a `message.id`, track it
2. Find orphaned tool results (user messages with `tool_result` content that reference tool calls in assistant messages not on the main chain)
3. Reattach orphaned tool results if needed

### Step 5: Strip internal fields — `Yt1(messages)`
**Function**: `Yt1`
**Location**: ~line 412696

Removes fields not needed by the API:
```javascript
function Yt1(A) {
  return A.map((q) => {
    let { isSidechain: K, parentUuid: _, ...Y } = q;
    return Y;
  });
}
```

Strips: `isSidechain`, `parentUuid`

## 4. Compaction System

### Compact Boundary
A compact boundary is a `system` message with `subtype: "compact_boundary"`:
```javascript
function of(A) {
  return A?.type === "system" && A.subtype === "compact_boundary";
}
```

### Compaction Result Structure
```typescript
interface CompactionResult {
  boundaryMarker: Message;          // the compact_boundary system message
  summaryMessages: Message[];       // summary of compacted content
  messagesToKeep?: Message[];       // preserved messages (not summarized)
  attachments: Message[];           // attachments to preserve across compaction
  hookResults: Message[];           // PostCompact hook results
  userDisplayMessage?: string;      // optional display text
  preCompactTokenCount: number;
  postCompactTokenCount: number;
  truePostCompactTokenCount: number;
  compactionUsage?: Usage;
}
```

### Context rebuild after compaction — `yl()`
```javascript
function yl(A) {
  return [A.boundaryMarker, ...A.summaryMessages, ...A.messagesToKeep ?? [], ...A.attachments, ...A.hookResults];
}
```

### Summary storage
Summaries are stored as separate JSONL entries: `{ type: "summary", leafUuid: string, summary: string }`
The `leafUuid` appears to be the UUID of the compact boundary marker (the last entry before compaction).

### Parent-child relationship
- `parentSessionId` set in global state when subagent is created

### Considerations for duncan

- Subagent sessions are in subdirectories of the main session, making discovery straightforward

## 8. Content Replacements

### Structure
- `type: "content-replacement"` entries with `sessionId` or `agentId` and `replacements` array
- Applied during session loading to modify message content
- May replace tool results, file contents, etc.

### Relevance for duncan
Content replacements modify what the model sees. Duncan should apply these to match what the original session saw.

## 9. Attachments

### How attachments work in CC
- `type: "attachment"` messages in the transcript
- Preserved across compaction (in `compactionResult.attachments`)
- Include file contents, images, etc. that were explicitly attached

### Relevance for duncan
Attachments are part of the context the model saw. Duncan should include them in the window.

## 12. Differences from Pi Duncan

| Aspect | Pi Duncan | CC Duncan |
|--------|-----------|-----------|
| Session format | Typed entries (`message`, `compaction`, `model_change`, etc.) | Flat messages (`user`, `assistant`, `system`, `progress`, `attachment`) + metadata entries |
| ID fields | `id` / `parentId` | `uuid` / `parentUuid` |
| Compaction | Inline `summary` + `firstKeptEntryId` in compaction entry | Separate `summary` entries + `compact_boundary` system message + `preservedSegment` relinking |
| Parent sessions | Explicit `parentSession` in session header | Not in session file; directory structure for subagents |
| System prompt | Stored/reconstructable from settings | Constructed at runtime from CLAUDE.md, MEMORY.md, settings |
| Message→API | `buildSessionContext()` → `convertToLlm()` → `streamSimple()` | `G26()` → `wHY()` → `Vs6()` → `OHY()` → `Yt1()` → `messages.create()` |
| Extension system | `pi.registerTool()` / `pi.registerCommand()` | MCP server (no extension API) |
| Session discovery | Session dir per hashed cwd, explicit parent links | Session dir per hashed cwd, temporal ordering |
